Operational Lessons from Benchmarking Real-Time Push Protocols Under CPU Limits

Protocol choices look equivalent in happy-path tests. Under cgroup pressure, event-loop contention, queue growth, and reconnection dynamics separate resilient designs from fragile ones.

By Harshit Singhal | March 3, 2026

1. Executive Framing

Benchmarking push systems under CPU limits should answer one question: what fails first when concurrency rises and pod headroom disappears? Median latency is usually stable too long to be useful. Tail latency and memory pressure expose instability earlier.

The most costly mistake is optimizing for maximum throughput while ignoring p95 and p99 delivery behavior during throttling and reconnect storms.

2. Runtime Architecture and Failure Mechanics

In async Python services, push delivery is bounded by event-loop scheduling fairness and per-connection write-backpressure. Under CPU throttling, cooperative tasks lose cadence; heartbeats, flush loops, and reconnection handlers compete for loop time.

  • Connection lifecycle: setup, steady-state writes, slow-consumer detection, and teardown must remain explicit.
  • Event-loop behavior: small scheduling delays accumulate into p99 spikes when thousands of connections share one loop.
  • Memory model: bounded queues cap resident memory; unbounded buffers turn transient bursts into prolonged instability.

These effects are directional and repeatable across stacks, even when absolute numbers differ by runtime or host profile.

3. Resource Cost Analysis: CPU, Memory, Connection State

CPU and memory costs are coupled through queueing behavior. As CPU availability tightens, drain rate drops. If ingress is unchanged, queue depth grows and memory pressure rises, which then increases GC or allocator overhead and worsens latency.

  • CPU-bound phase: serialization and network write scheduling dominate.
  • Memory-bound phase: buffered messages and per-connection state dominate.
  • State overhead: protocols with heavier lifecycle bookkeeping amplify both costs at high connection counts.

In containerized environments, this transition can happen quickly because cgroup limits make backlog growth visible as OOM risk instead of a slow degradation.

4. Behavior Under Scale

Scale failures are usually phased.

  • First phase: p95 drift with stable median latency.
  • Second phase: p99 expansion and reconnect churn.
  • Third phase: memory instability, write timeouts, and partial collapse in selected pods.

Autoscaling helps only if metrics capture these phases. Scaling on request rate alone often reacts too late.

5. Operational Risks and When Not to Use This Approach

A benchmarking strategy centered on one-way push behavior is inappropriate when product requirements are truly duplex and low-latency in both directions.

  • Do not generalize one-way benchmark conclusions to interactive bidirectional workloads.
  • Do not run protocol tests without reconnection and slow-consumer scenarios.
  • Do not rely on averages; tail-latency and drop behavior must be primary signals.
  • Do not keep queue depth unbounded to preserve nominal throughput during bursts.

6. Decision Matrix

7. Monitoring and SLO Implications

Push workloads need saturation-aware observability.

  • Latency: track delivery p50, p95, and p99, segmented by protocol and pod.
  • Backpressure: queue depth, enqueue drops, write timeout rate, and slow-consumer counts.
  • Connection health: active sockets, reconnect rate, disconnect reasons, handshake failures.
  • Container pressure: memory working set, OOM events, CPU throttled time, and restart loops.

SLO policy should explicitly encode degradation behavior: acceptable drop strategy, max reconnect churn, and escalation conditions when p99 rises with queue depth.

Minimal Async Pattern for Bounded Delivery Under CPU Pressure

This pattern favors bounded memory and controlled shedding instead of unbounded buffering. 

import asyncio
import time
from collections import deque
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse

app = FastAPI()

class DeliveryState:
    def __init__(self, max_queue: int = 512):
        self.q = asyncio.Queue(maxsize=max_queue)
        self.dropped = 0
        self.delivery_ms = deque(maxlen=2048)

state = DeliveryState()

async def publish(event: str) -> None:
    if state.q.full():
        _ = state.q.get_nowait()  # Drop oldest to cap memory.
        state.dropped += 1
    state.q.put_nowait((time.perf_counter(), event))

async def stream(request: Request):
    while True:
        if await request.is_disconnected():
            break
        ts, event = await state.q.get()
        state.delivery_ms.append((time.perf_counter() - ts) * 1000)
        yield f"data: {event}\\n\\n"

@app.get("/events")
async def events(request: Request):
    return StreamingResponse(stream(request), media_type="text/event-stream")

8. Engineering Conclusion

Benchmarking under CPU limits is only useful if it captures tail behavior, memory stability, and reconnection dynamics. Systems that look efficient at median latency can still fail operationally through p99 expansion and memory-driven instability.

Treat protocol choice as part of runtime control strategy. Prefer architectures that make saturation visible, keep buffers bounded, and recover predictably after pressure drops.
